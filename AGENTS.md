#### 二、强制硬性要求，必须严格执行

1. **编辑规则**：只在文件原有内容基础上做局部修改，**禁止重写、禁止覆盖整个文件**，禁止删除无关内容。

3. **代码保护**：不得改动任何无关的原有代码逻辑，只修改指定内容。
4. **可还原**：所有修改必须支持**撤销 / 还原至上一次版本**。

6. **命令执行与转义安全**：涉及命令嵌套、路径、变量、引号或结构化文本时，优先减少解析层级，当前 Shell 已经是 PowerShell 时，禁止无必要地再嵌套 `powershell -Command "..."`，若必须嵌套，包含 `$变量` 的内层脚本必须用单引号包裹，禁止双引号导致外层提前展开变量，复杂 JSON、GeoJSON、MCP 参数和深层数组结构必须使用原始 JSON、临时 JSON 文件或 Node.js `JSON.stringify()`，并在发送前打印核对最终内容。
7. **RTK 命令行输出压缩**：@C:\Users\xzq\.codex\RTK.md
      - 所有 shell 命令默认优先使用 `rtk` 前缀；优先使用 RTK 原生命令，例如 `rtk read`、`rtk rg`、`rtk git`。
      - PowerShell cmdlet 不得直接使用 `rtk Get-Content`、`rtk Get-ChildItem`、`rtk Remove-Item` 等形式，也不得通过 `rtk run` 或 `rtk { ... }` 执行。
      - 若必须执行 PowerShell cmdlet，统一使用：
        `rtk proxy pwsh -NoProfile -Command "<PowerShell 命令>"`
        若 `pwsh` 不可用，再使用：
        `rtk proxy powershell -NoProfile -Command "<PowerShell 命令>"`
      - 在 Windows / PowerShell 环境中，不推荐使用 `rtk ls`；目录列举优先使用 `rtk rg --files <目录>`，若必须读取目录项，使用上述 `rtk proxy pwsh ... Get-ChildItem ...` 形式。
      - Windows 下不要把带 `*` 的文件名模式直接作为 `rg` 的路径参数；查找配置文件优先使用：
        `rtk rg --files | rtk rg "<文件名正则>"`
      - 不使用 `rtk rewrite` 处理 PowerShell cmdlet；PowerShell cmdlet 统一按上述 `rtk proxy pwsh ...` 规则执行。
8. **代码与文件搜索**：搜索文件名、代码内容或目录内容时，必须优先使用 `rg` / `rg --files`（启用 RTK 时为 `rtk rg`），禁止绕过 `rg` 直接使用 `findstr`、`Select-String`、`Get-ChildItem -Recurse` 做同类搜索，除非 `rg` 不存在、执行失败或明确不适用。
9. **文件与行号引用格式：**
   - 当回复中提到文件、代码位置、报错位置、修改点、调用点或需要用户查看具体实现时，必须使用 Markdown link 的本地绝对路径形式。
   - 格式统一为：
     - `[文件名](/绝对路径/文件名)`
     - `[文件名:行号](/绝对路径/文件名:行号)`
     - `[文件名:行号:列号](/绝对路径/文件名:行号:列号)`
   - 不要使用相对路径。
   - 不要只写纯文本行号，例如 `src/main.cpp:12`。
   - 不要使用 `file://`、`vscode://` 或网页链接代替本地绝对路径。
   - 当只需要提示文件位置时，使用文件链接。
   - 当需要精确定位问题时，使用带行号的链接。
   - 示例：
     - `[AGENTS.md](/h:/code/RenderingPipeline/AGENTS.md)`
     - `[AGENTS.md:111](/h:/code/RenderingPipeline/AGENTS.md:111)`
     - `[TriangleScene.cpp:42](/h:/code/RenderingPipeline/src/scenes/TriangleScene.cpp:42)`
   - 其他相关的内容提示也都以用户易读的角度出发，做成相关的 Markdown 格式。

#### 四、Apifox MCP 调用规范

##### 一、触发判断总原则

本次任务只要涉及真实后端接口，必须先判断改动类型，再决定是否触发 Apifox MCP 流程。

1. **无条件触发**

   出现以下任一情况，必须立即触发 Apifox MCP 流程：

   1. 用户明确要求调用、查询、读取、刷新、核对 Apifox。
   2. 用户明确要求读取或核对接口文档、OpenAPI、OAS、接口定义、接口参数、请求体、响应结构。
   3. 用户要求确认接口字段、字段类型、必填项、枚举、字典、分页字段或 `$ref` 引用。

2. **需要先判断是否字段变更**

   出现以下任一情况，必须先判断是否涉及新的请求字段、响应字段或字段映射：

   1. 用户明确提及要使用的接口名称、接口关键词、接口用途或接口路径。
   2. 用户要求复用项目已有 API 文件或 API 方法。
   3. 生成的代码会调用真实后端接口。
   4. 页面功能涉及提交、保存、查询、分页、详情、导出、上传、删除等接口交互。

3. **必须触发**

   在 `2` 的场景中，只要出现以下任一情况，必须触发 Apifox MCP 流程：

   1. 新增、删除、修改任何请求字段。
   2. 新增、删除、修改任何响应字段。
   3. 新增、删除、修改 UI 字段与接口字段的映射。
   4. 新增字段需要提交给后端。
   5. 新增字段需要从后端读取展示。
   6. 修改请求参数结构、提交结构或响应结构。
   7. 无法确认是否涉及接口字段新增或变更。

4. **可不触发**

   在 `2` 的场景中，如果同时满足以下条件，可不触发 Apifox MCP，按项目现有代码直接调整：

   1. 只是复用项目已有接口封装。
   2. 不新增、不删除、不修改请求字段。
   3. 不新增、不删除、不修改响应字段。
   4. 不新增、不删除、不修改 UI-API 字段映射。
   5. 不改变提交结构、查询参数、分页参数、响应解析逻辑。



##### 二、字段变更判断

以下情况视为“涉及接口字段新增或变更”，必须触发 Apifox MCP：

- 新增表单字段并提交给后端。
- 新增页面字段并从接口响应中读取。
- 新增筛选条件、分页参数、排序参数、导出参数、上传参数。
- 修改请求体结构、请求字段名、字段类型、字段枚举、字段默认值。
- 修改响应字段展示、响应字段映射、字典 / 枚举解释。
- 新增或修改 API 方法、接口路径、请求方法。
- 复用现有接口，但本次新增了原代码中没有提交过的字段。
- 根据 MasterGo / 截图 / 口头描述，把新的 UI 字段绑定到接口字段。

以下情况可不触发 Apifox MCP：

- 仅调整样式、布局、顺序、显隐、文案，不改接口字段。
- 仅复用已有 API 方法，且请求参数和响应字段完全沿用原逻辑。
- 仅移动已有字段位置，不新增字段、不改字段含义。
- 仅修复前端交互状态，不影响接口入参和出参。



##### 三、禁止用现有代码替代接口文档

项目内已有 API 文件、历史调用代码、mock 数据、类型定义、注释说明，只能用于判断“是否已有接口封装”和“是否新增字段”。

一旦判定涉及接口字段新增或变更，必须以刷新后的 Apifox / OAS 文档为准，不得用现有代码猜接口参数。



##### 四、首次文档确认

当前会话第一次触发 Apifox MCP 时，必须先停止后续操作，列出所有可用 Apifox 文档。

展示固定格式为：

```markdown
已检测到 Apifox/OAS MCP 工具已连接，可用文档如下：

1. **文档标题**
   MCP 命名空间：xxx
   项目 / 模块说明：xxx
   
2. **文档标题**
   MCP 命名空间：xxx
   项目 / 模块说明：xxx
```

展示规则：

1. 必须使用固定的普通 Markdown 展示形式，禁止使用代码块或表格展示可用文档。
2. 每个文档必须使用有序列表展示。
3. 文档标题必须加粗。
4. 每个文档必须包含文档标题、MCP 命名空间、项目 / 模块说明。
5. MCP 命名空间必须使用行内代码格式。

用户确认目标文档前，禁止读取具体接口，禁止新增、修改、删除业务代码、类型文件、API 文件、mock 文件或路由文件。



##### 五、文档选择缓存

用户确认目标文档后，本会话后续默认使用该文档，除非用户明确要求切换文档，否则不得自行改用其他 Apifox 文档。



##### 六、刷新优先

每次读取已选文档的接口信息前，必须先调用该文档对应的刷新工具，如果用户说明 `接口有更新`、`重新修改`、`重新读接口`、`字段变了`，必须重新刷新文档，重新列候选接口，重新确认接口和字段映射。



##### 七、候选接口确认

触发 Apifox 后，如果用户已提到接口名称、接口关键词、接口用途或接口路径，刷新文档后必须先列出候选接口。

使用固定的普通 Markdown 展示形式，候选接口展示格式统一为 ：

【一级中文分组名称】/【二级中文分组名称】/【...】-接口名称（METHOD PATH）

展示要求：

1. 中文分组名称必须从当前接口自身详情的 tags 字段中读取，且必须加粗展示，禁止拆分和简化，必须完整被【】包裹。
2. 存在同名、相似名称、相似路径、同用途接口时，必须全部列出，不得自行二选一。
3. 用户确认具体接口前，禁止读取接口详情以外的业务生成依据，禁止修改业务代码。



##### 八、未指定接口时的处理

触发 Apifox MCP 调用规范，不代表可以直接读取某个具体接口。

如果用户没有明确提及接口名称、接口关键词、接口用途、接口路径，且无法从用户指定的项目 API 方法中读取明确请求路径，则不得猜测接口，不得读取具体接口详情，此时生成代码所需数据使用静态数据，并在回复中说明：当前未指定接口，已按静态数据处理。



##### 九、项目 API 方法的特殊处理

如果用户指定了项目内 API 文件或 API 方法，例如 src/api/xxx.js 的 saveXxx：

1. 允许先只读该 API 文件，确认请求方法和请求路径。
2. 如果本次不涉及新增或变更接口字段，可直接复用现有 API 方法调整代码。
3. 如果本次涉及新增或变更接口字段，必须用读取到的请求路径 / 方法进入 Apifox 候选接口确认流程。
4. 项目 API 文件不得替代 Apifox 接口详情。



##### 十、 接口详情读取要求

用户确认具体接口后，必须读取并核对：

- 请求方法
- 请求路径
- Header 参数
- Path 参数
- Query 参数
- Request Body
- Content-Type
- 请求字段类型
- 必填字段
- 默认值
- 枚举 / 字典字段
- 分页字段
- 响应结构
- 响应字段类型
- $ref 引用内容

如存在 $ref，必须继续读取引用资源，不得只看第一层接口定义。



##### 十一、UI-API Binding Map 字段确认

只要本次基于接口新增或变更 UI 字段、提交字段、响应字段展示，必须先输出 UI-API Binding Map，等待用户确认后才能写入业务代码。

UI-API Binding Map 输出结构：

```text
当前接口：xxx
1. 视觉标题：xxx
   疑似匹配字段：xxx
2. 视觉标题：xxx
   疑似匹配字段：xxx
```

即使中文标题与接口字段 title 完全一致，也必须先确认字段映射。

若该字段用于固定下拉选项，则根据接口字段中文描述给出建议映射，例如：

```text
视觉标题：性别-男/女
疑似匹配字段：gender
建议映射：男 = 1，女 = 2
```



##### 十二、代码生成门禁

只有同时满足以下条件，才能基于新增或变更接口字段生成代码：

1. 已确认 Apifox 文档。
2. 已刷新文档。
3. 已列出候选接口。
4. 用户已确认具体接口。
5. 已读取接口详情和 $ref。
6. 已输出 UI-API Binding Map。
7. 用户已确认字段映射。

未完成以上任一项，禁止新增、修改、删除业务代码、API 文件、类型文件、mock 文件或路由文件。



##### 十三、Mock 固定静态数据

凡是本次代码生成涉及真实后端接口，必须先补齐 mock 数据后再收尾，确保生成或者更新固定 mock 数据。

mock 数据字段必须以已确认接口详情为准。

生成结构，参考《七、组件封装与代码生成规范》下的《二、目录结构规范》。

涉及图片字段时，从 https://unsplash.com/ 随机选取图片使用。



##### 十四、歧义处理

出现以下任一情况，必须停止并等待用户确认：

- 候选接口过多。
- 接口名称重复。
- 接口路径相似。
- 字段类型缺失。
- 请求 / 响应结构缺失。
- $ref 读取失败。
- 同一 UI 字段可映射多个接口字段。
- 项目已有代码与 Apifox 文档不一致。



##### 十五、收尾说明

完成后必须简要说明：

- 是否触发 Apifox MCP。
- 如触发：使用的 Apifox 文档、确认的接口、请求方法、请求路径、关键字段。
- 如未触发：说明未新增或变更接口字段，按现有接口封装调整。
- 是否生成 / 更新 mock 数据。
- 改动涉及的文件。



#### 五、MasterGo MCP 图层强制规则

1. **MasterGo 读取门禁**
   - 凡是用户提供 MasterGo 链接、fileId、layerId、截图还原要求或明确要求“一比一还原”，必须先调用 MasterGo MCP 读取目标图层。
   - 未成功读取 MasterGo 图层前，禁止根据记忆、截图猜测或项目常见样式直接生成代码。
   - 若 MasterGo MCP 读取失败，必须停止并说明失败原因，等待用户确认是否改用截图或静态描述继续。

2. **图层类型识别**
   - 调用 MasterGo MCP 读取图层时，当目标图层名匹配 `uuid-*`，必须将其视为页面局部组件。
   - `uuid-*` 图层生成的组件根节点必须绑定 `masterGoUuid="图层名"`。
   - 若目标图层不是 `uuid-*`，仍需判断其是否包含多个独立功能区；若包含多个功能区，必须结合「页面主文件职责」和「局部组件强制拆分条件」拆分为页面局部组件。

3. **生成前元素清单**
   - 生成代码前，必须先提取并输出 MasterGo 界面元素 / 控件清单。
   - 清单必须包含：
     1. 页面模块或区块名称。
     2. 文本内容。
     3. 可交互控件，例如按钮、输入框、下拉框、图片预览、展开 / 收起、提交等。
     4. 表格、信息栅格、列表、图片区等重复结构。
     5. 图标、图片、背景图、分割线、状态标识等视觉元素。
   - 未输出元素 / 控件清单前，禁止新增或修改业务页面代码。

4. **禁止自行补充设计稿不存在的元素**
   - 未出现在 MasterGo 图层中的界面元素 / 控件，禁止自行添加。
   - 即使项目中已有常见组合，只要 MasterGo 图层没有对应节点，也不得自动补充。
   - 若业务逻辑确实需要额外控件或状态，必须先向用户说明“设计稿中不存在该界面元素 / 控件”，并等待用户确认后才能添加。
   - 用户未确认前，禁止把额外控件写入代码。

5. **交互控件语义映射**
   - 可交互元素必须优先映射为项目组件库或当前 UI 框架组件，再进行样式覆盖。
   - 禁止无视语义地把按钮、下拉框、输入框、图片预览、表单项等控件写成纯静态 `div`。
   - 若控件样式与组件库默认样式不同，应保留组件语义，通过 class、CSS 变量或局部样式覆盖视觉。
   - 若 MasterGo 视觉样式与项目现有全局组件或业务组件能力重合，必须优先复用或参考项目组件。

6. **页面模块拆分联动**
   - MasterGo 图层中出现 2 个及以上独立功能模块时，必须拆分为页面局部组件。
   - 功能模块包括但不限于：信息展示区、表单区、处置区、详情区、表格区、图片区、地图区、视频区、弹窗区、抽屉区。
   - 页面主文件 `index.vue` 只能负责路由参数、接口调用、顶层 loading、模块数据组装和事件转发。
   - 禁止把 MasterGo 中多个功能模块的大段模板、样式和交互逻辑全部堆在 `index.vue` 中。

7. **视觉规律识别**
   - 生成代码前，必须识别 MasterGo 中重复结构的视觉规律。
   - 对表格、信息栅格、列表、详情字段矩阵、图片区等结构，必须明确：
     1. 一行包含几个单元。
     2. 背景色是按整行、整列、单元格、分组还是状态交替。
     3. label 和 value 是否共用同一背景块。
     4. 是否存在跨列、空白占位、固定宽度列或末行不满行。
     5. 间距、边框、圆角、分割线是否按行或按块出现。
   - 多列 grid 禁止直接使用 `nth-child(odd/even)` 猜测交替背景。
   - 若需要实现行背景，必须通过行索引、分组数据或明确的整行选择器实现。
   - 差异自检时必须说明这些视觉规律是否已对齐设计稿。

8. **图片与静态资源**
   - 涉及图片、图标、背景图等静态资源时，必须自动切图并存放至对应页面文件夹的 `assets` 目录。
   - 资源按功能分文件夹隔离，统一路径格式：
     `src/views/页面模块/assets/分类/*`
   - 禁止把页面私有图片放入全局公共资源目录，除非明确具备全局复用价值。

##### 三、组件封装规范

无论是全局通用组件还是页面局部业务组件的封装，都遵循以下细则：

1. **基础原则**
   - 所有自定义表单类、筛选类、选择类、弹窗类、抽屉类、上传类、配置面板类组件，必须优先支持 `v-model` 双向绑定语法。
   - 组件封装必须遵循**单一职责原则**：一个组件只负责一个明确的业务能力，禁止把多个无强关联的业务逻辑堆在同一个组件内。
   - 组件必须具备可维护性、可扩展性和可读性，避免过度封装、过度抽象和无意义的通用化。
   - 组件内部禁止直接修改 `props` 传入的任何值，包括对象、数组的深层属性。
   - 若确实需要修改对象或数组数据，必须先在组件内部创建副本，再通过事件向外通知父组件更新。
   - Props 数量控制在合理范围，不超过 8~10 个，复杂组件应拆分为子组件。
   - 即使是业务组件，也需具备一定程度的可复用性，不写死硬编码业务数据。
2. **v-model 封装规范**
   - Vue 2 项目中，默认使用 `value` + `input` 实现 `v-model`。
   - Vue 3 项目中，默认使用 `modelValue` + `update:modelValue` 实现 `v-model`。
   - 必须使用 Vue **计算属性（computed）**，通过 `get()` 和 `set()` 方法进行数据拦截与处理。
   - Vue 2 写法中，计算属性 `get` 返回 `this.value`；`set` 中通过 `this.$emit('input', 新值)` 触发更新。
   - Vue 3 写法中，计算属性 `get` 返回 `props.modelValue`；`set` 中通过 `emit('update:modelValue', 新值)` 触发更新。
   - 禁止在组件内部出现 `this.value = xxx`、`props.modelValue = xxx`、`props.xxx = xxx` 等直接赋值行为。
3. **Props 设计规范**
   - 所有 `props` 必须声明类型、默认值和用途说明，并提供清晰的类型校验（String/Number/Boolean/Array/Object/Function）。
   - 布尔类型 `props` 命名必须使用具备明确语义的前缀，例如 `showXxx`、`enableXxx`、`disabledXxx`、`readonlyXxx`。
   - 数组和对象类型 `props` 的默认值必须使用函数返回：`default: () => ({})` / `default: () => []`，禁止直接写 `{}` 或 `[]`。
   - 组件 `props` 只负责接收外部配置，禁止在组件内部把 `props` 当作内部状态直接修改。
   - 若某个 `props` 只服务当前业务页面，不具备通用意义，不得强行放入通用组件 API 中。
4. **事件设计规范**
   - 组件对外事件必须语义清晰，禁止使用 `handleXxx`、`onXxx` 作为事件名。
   - 推荐事件命名：`change`、`confirm`、`cancel`、`submit`、`reset`、`select`、`remove`、`clear`、`visible-change`。
   - 所有会改变父级数据的操作，必须通过 `$emit` 或 `emit` 通知父组件，由父组件决定是否更新数据。
   - 事件参数必须保持稳定，禁止同一个事件在不同场景下返回完全不同的数据结构。
   - 复杂事件参数必须使用对象形式返回，并在注释或文档中说明字段含义。
5. **插槽设计规范**
   - 组件中存在可自定义区域时，必须优先考虑使用 `slot`，避免通过大量 `props` 控制复杂 UI。
   - 常用插槽命名建议：`default`、`header`、`footer`、`title`、`extra`、`action`、`empty`、`item`。
   - 作用域插槽必须明确暴露字段名称，禁止暴露未经整理的内部完整对象，避免外部依赖组件内部实现。
6. **组件状态管理规范**
   - 组件内部状态必须与外部绑定值区分清楚，禁止混用。
   - 临时交互状态，例如 loading、hover、active、expanded，可在组件内部维护。
   - 业务结果状态，例如选中值、表单值、筛选条件、弹窗显示状态，必须通过 `v-model` 或事件交由父组件维护。
   - 组件内部如需根据 `props` 初始化状态，必须考虑后续 `props` 变化的同步逻辑。
7. **TypeScript 类型规范**
   - 新生成的 Vue 组件若项目支持 TypeScript，必须为 `props`、`emit`、插槽参数、内部方法参数和返回值补充类型声明。
   - 复杂数据结构必须单独声明 `interface` 或 `type`，禁止在代码中反复书写大段内联类型。
   - 对外暴露的类型建议统一导出，便于业务页面复用。
8. **组件方法暴露规范**
   - 默认不向外暴露组件内部方法。
   - 只有当父组件确实需要主动触发组件行为时，才允许暴露方法，例如 `open()`、`close()`、`reset()`、`validate()`、`clear()`。
   - Vue 3 中暴露方法必须使用 `defineExpose`，并添加清晰注释。
   - Vue 2 中通过 `ref` 调用的方法必须保持简单稳定，禁止让父组件依赖过多内部细节。
9. **样式封装规范**
   - 组件样式必须尽量使用局部作用域，避免污染全局样式。
   - 样式类名必须具备组件语义，禁止使用 `box`、`left`、`right`、`text` 等过于宽泛的类名。
   - 通用组件禁止写死业务页面宽度、高度、颜色和定位；确需定制时，应通过 `props`、CSS 变量或外层容器控制。
   - 组件内部禁止随意覆盖第三方组件库的全局样式；如必须覆盖，需限定作用域并添加注释说明原因。
10. **组件文档规范**
    - 可复用组件必须提供使用示例、`Props` 说明、事件说明、插槽说明和常见使用场景。
    - 若组件支持 `v-model`，文档必须明确绑定字段类型、默认值和更新时机。
    - 组件存在复杂交互时，必须补充完整代码示例，不能只给单行调用示例。
11. **组件质量要求**
    - 组件必须保持主结构清晰，模板层避免出现过长的复杂表达式。
    - 复杂计算逻辑必须抽离为 `computed`、方法或独立工具函数。
    - 重复出现两次以上的结构或逻辑，应评估是否抽离为局部子组件或工具函数。
    - 通用组件不得强依赖具体业务接口、具体页面路由、具体 Pinia/Vuex 模块或固定后端字段。

4. **局部组件强制拆分条件**
      - 页面存在 2 个及以上功能模块时，必须拆分为页面局部组件。
      - 功能模块包括但不限于：信息展示区、查询区、表单区、处置区、详情区、图片区、表格区、弹窗区、抽屉区、地图区、视频区。
      - 以下任一情况出现，也必须拆分为页面局部组件：
        1. `<template>` 中出现 2 个及以上业务卡片、业务区块或模块标题。
        2. 单个模块拥有独立的表单、列表、详情、图片预览、提交、校验或状态切换逻辑。
        3. 单个模块拥有独立样式区域，且样式类名可以用业务语义命名。
        4. 本次新增后 `index.vue` 明显超过容器职责，或新增代码超过 200 行。
      - 若因极小改动暂不拆分，必须在写代码前说明原因，并等待用户确认。

5. **局部组件目录要求**
      - 页面局部组件统一放在当前页面目录下的 `components/` 文件夹中。
      - 局部组件必须使用大驼峰命名，并体现业务语义，例如：
        - `WarnInfoCard.vue`
        - `WarnDisposeSection.vue`
      - 与局部组件强相关的私有类型、工具函数、样式和资源，应优先收敛在当前页面目录内，禁止无必要扩散到全局目录。

6. **局部组件职责边界**
      - 局部组件只接收当前模块需要的 `props`，禁止把整页详情对象无差别传入，除非该模块确实需要完整对象。
      - 局部组件内部禁止直接调用与页面无关的全局业务接口；接口调用默认由 `index.vue` 或页面业务 hook 统一协调。
      - 局部组件通过语义化事件向父级通信，例如 `submit`、`refresh`、`change`、`preview`，禁止使用含糊事件名。
      - 可编辑表单类局部组件必须优先支持 `v-model` 或明确的 `modelValue` / `update:modelValue` 机制。

7. **全局组件升级判断**
      - 生成局部组件前，必须判断该模块是否具备全局复用价值。
      - 满足以下任一条件，视为具备全局复用价值：
        1. 当前项目已有 2 处及以上相似使用场景。
        2. 属于基础组件范畴，例如通用表单项、通用选择器、通用上传、通用图片预览、通用弹窗。
        3. 未来可明确预见会被多个业务页面复用。
      - 不具备全局复用价值的，只允许作为页面局部组件实现，禁止过度抽象为全局组件。

8. **全局组件升级流程**
      - 若判断具备全局复用价值，必须先核查项目内 `src/components` 目录是否已有可复用组件。
      - 若已有可复用组件，必须优先复用或基于现有组件扩展，禁止重复造轮子。
      - 若项目内没有合适组件，必须先询问用户是否从「组件沉淀目录」查找相似组件。
      - 只有用户确认后，才能执行组件沉淀目录前置检查。
      - 未经用户确认，禁止访问、复制、修改或初始化组件沉淀目录。

9. **组件沉淀目录使用流程**
      - 用户确认从组件沉淀目录查找后，必须严格执行组件沉淀目录前置检查。
      - 若选取到合适组件，只允许复制目标组件的真实组件英文命名文件夹到项目 `src/components` 目录，再基于副本改造。
      - 改造完成后，如仍具备全局复用价值，必须主动询问用户是否沉淀并覆盖组件沉淀目录旧版本。
      - 未经用户确认，禁止自动沉淀、覆盖或提交组件沉淀目录内容。

10. **无可用组件时的处理**

      - 若项目现有组件和组件沉淀目录均无合适组件，必须暂停实现。
      - 暂停时需要说明：
        1. 该能力为什么具备全局复用价值。
        2. 建议抽离的组件名称。
        3. 建议的 `props`、事件、插槽和使用场景。
      - 用户确认后，才能继续封装全局组件。
       - 若用户不确认抽离为全局组件，则只能作为页面局部组件实现。

11. **生成后自检**

       - 页面代码生成后，必须检查 `index.vue` 是否仍然符合容器职责。
       - 若 `index.vue` 中仍存在多个业务模块的大段模板、样式或交互逻辑，必须继续拆分，不能以实现已完成为由跳过。
       - 收尾说明中必须简要列出：
         1. 本次拆分出的局部组件。
         2. 未抽为全局组件的原因。
         3. `index.vue` 当前保留的职责。



##### 五、代码注释规范

1. 所有新生成或修改的代码，必须**尽可能添加详细、清晰的注释**。

2. 只有本次改动的代码（仅新生成的部分）直接涉及以下第三方技术栈：**Openlayers、Vben、Cesium**，必须**立即暂停当前操作**，并主动向用户询问：

   **“检测到代码涉及第三方技术栈，是否需要生成超详细注释？”**

3. 只有在用户**手动确认需要详细注释**后，才继续生成代码。

4. 生成注释时，**必须按照零基础新手标准**（假设用户完全未使用过该技术栈），对**每一个配置项、每一段关键逻辑、每一个属性**的**作用、含义、使用方式**进行逐行详细解释。



#### 八、删除操作规范

1. 严格禁止递归删除，任何情况下不得执行：
   - `del /s`
   - `rd /s`
   - `rmdir /s`
   - `Remove-Item -Recurse`
   - `rm -rf`

2. 禁止通配符删除和模糊路径删除：
   - 禁止使用 `*`、`?` 删除文件或目录。
   - 删除前必须展示最终解析后的完整绝对路径。

3. 单个文件删除：
   - 允许删除单个明确文件。
   - 必须使用：
     `Remove-Item -LiteralPath "完整文件路径"`

4. 空文件夹删除：
   - 允许删除空文件夹。
   - 删除前必须逐个向用户确认，确认内容包括：
     - 文件夹完整路径
     - 确认该目录为空
     - 即将执行的命令
   - 用户确认后，只允许执行：
     `Remove-Item -LiteralPath "完整文件夹路径"`
   - 禁止添加 `-Recurse`。

5. 非空文件夹删除：
   - 禁止直接删除非空文件夹。
   - 必须先使用 `rg --files "目标目录"` 列出内部文件。
   - 向用户展示文件清单并获得确认后，逐个删除文件。
   - 文件删完后，再按“空文件夹删除”规则逐个删除空目录。

6. 组件迁移 / 目录改名后的空目录：
   - 当移动文件后导致源目录变为空目录时，也视为“删除文件夹相关场景”。
   - 必须主动提示用户是否删除该空目录。
   - 未确认前不得删除空目录。

7. 安全边界：
   - 删除目标必须位于当前工作区或用户明确指定目录内。
   - 路径不明确、目录非空但用户未确认文件清单、或命令需要递归删除时，必须停止操作。



#### 九、Context7 调用规则

Use Context7 MCP to fetch current documentation whenever the user asks about a library, framework, SDK, API, CLI tool, or cloud service -- even well-known ones like React, Next.js, Prisma, Express, Tailwind, Django, or Spring Boot. This includes API syntax, configuration, version migration, library-specific debugging, setup instructions, and CLI tool usage. Use even when you think you know the answer -- your training data may not reflect recent changes. Prefer this over web search for library docs.

Do not use for: refactoring, writing scripts from scratch, debugging business logic, code review, or general programming concepts.

Steps：

1. Always start by checking the current conversation cache for a matching Context7 library ID. If a matching library ID for the same library, same major version, and same documentation source is cached, reuse it directly and skip `resolve-library-id`. If no matching library ID is cached, call `resolve-library-id` using the library name and the user's question, unless the user provides an exact library ID in `/org/project` format
2. Pick the best match (ID format: `/org/project`) by: exact name match, description relevance, code snippet count, source reputation (High/Medium preferred), and benchmark score (higher is better). If results don't look right, try alternate names or queries (e.g., "next.js" not "nextjs", or rephrase the question). Use version-specific IDs when the user mentions a version
3. `query-docs` with the selected library ID and the user's full question (not single words)
4. Answer using the fetched docs



**通用约束：**

1. 自动调用 context7-mcp 时，需要获取与当前项目中 `package.json` 版本对应的官方文档。
2. 在回复开头明确说明：`已自动调用 context7-mcp 查询官方文档（xxx 版本）并基于文档实现`。
3. 严格按照查询到的文档、API、示例编写代码。
4. 禁止使用过时 API、禁止幻觉、禁止编造参数。
